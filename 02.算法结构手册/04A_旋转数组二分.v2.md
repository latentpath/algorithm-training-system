# 第四章补充：旋转数组二分结构

> 核心信念：**旋转数组二分不是恢复有序，而是在局部单调结构中判断答案所在半边。**

---

## ① 本章解决什么结构问题？

本章处理的是整体无序但局部仍有单调性的数组。

旋转数组来自一个有序数组的切分与拼接：

```text
[4, 5, 6, 7, 0, 1, 2]
```

它不是整体有序，但在任意一次二分中通常仍能判断：

- 左半边有序，或右半边有序。
- `mid` 位于大值段，或位于小值段。
- 答案是否可能落在某个局部有序半边。

本章的核心问题是：

> 当整体单调性被旋转破坏后，如何利用局部单调性继续安全排除一半候选。

---

## ② 本章的结构判断入口

### 结构识别信号

| 信号 | 可能结构 |
|---|---|
| 有序数组被旋转 | 局部单调二分 |
| 在旋转数组中找 `target` | 判断哪一半有序 |
| 找旋转数组最小值 | 判断 `mid` 属于大值段还是小值段 |
| 无重复元素 | 标准旋转二分 |
| 有重复元素 | 局部单调性可能退化 |

旋转数组二分不能只套标准二分。每次排除半边前，都要说明保留下来的半边仍覆盖答案。

---

## ③ 本章的结构骨架

旋转数组二分有两类问题：

| 问题 | 目标 | 边界处理 |
|---|---|---|
| 查找 `target` | 判断目标是否在某个有序半边 | `mid` 命中后可丢弃 |
| 查找最小值 | 收缩最小值所在区间 | `mid` 可能是答案，不能随意丢弃 |

二者虽然都叫旋转数组二分，但不变量不同。

---

## ④ 核心模型拆解

### 1）查找 target：先判断有序半边

#### 结构定义

在 `[left, right]` 中，至少有一侧保持有序。先判断哪一侧有序，再判断 `target` 是否落在该侧范围内。

#### 不变量

```text
[left, right] 始终覆盖 target 可能出现的位置
```

如果 `nums[mid] == target`，可以直接返回。否则 `mid` 不再是候选，因此边界更新可以使用 `mid + 1` 或 `mid - 1`。

#### 代码骨架

```java
int left = 0;
int right = nums.length - 1;

while (left <= right) {
    int mid = left + (right - left) / 2;

    if (nums[mid] == target) {
        return mid;
    }

    if (nums[mid] >= nums[left]) {
        // 左半边有序
        if (target >= nums[left] && target < nums[mid]) {
            right = mid - 1;
        } else {
            left = mid + 1;
        }
    } else {
        // 右半边有序
        if (target > nums[mid] && target <= nums[right]) {
            left = mid + 1;
        } else {
            right = mid - 1;
        }
    }
}

return -1;
```

#### 适用条件

- 原数组升序且被旋转。
- 元素无重复，或重复不影响判断有序半边。
- 目标是查找某个具体值。

#### 失效条件

- 大量重复导致 `nums[left] == nums[mid] == nums[right]`，无法判断哪边有序。
- 数组不是由有序数组旋转得到。
- 目标不是查找位置，而是找全局最值或边界。

有重复时通常需要退化处理，例如 `left++` 或 `right--`，最坏复杂度可能退化为 `O(n)`。

---

### 2）查找最小值：收缩最小值所在区间

#### 结构定义

找最小值时，目标不是 `target`，而是旋转断点。推荐用 `nums[right]` 判断 `mid` 位于哪一段：

- `nums[mid] > nums[right]`：`mid` 在左侧大值段，最小值在右侧。
- `nums[mid] <= nums[right]`：`mid` 在右侧小值段，`mid` 可能就是最小值。

#### 不变量

```text
[left, right] 始终包含最小值
```

当 `nums[mid] <= nums[right]` 时，不能写 `right = mid - 1`，因为 `mid` 可能就是答案。

#### 代码骨架

```java
int left = 0;
int right = nums.length - 1;

while (left < right) {
    int mid = left + (right - left) / 2;

    if (nums[mid] > nums[right]) {
        left = mid + 1;
    } else {
        right = mid;
    }
}

return nums[left];
```

#### 为什么用 `left < right`

循环中存在：

```java
right = mid;
```

当 `left == right` 时，`mid == right`，如果使用 `left <= right`，区间可能无法缩小。

#### 适用条件

- 数组由升序数组旋转而来。
- 目标是最小值或旋转点。
- 无重复时可以稳定 `O(log n)`。

#### 失效条件

- 有重复且 `nums[mid] == nums[right]`，无法判断最小值在哪一侧。
- 需要找最大值时，判定关系要重新定义。

---

### 3）有重复元素的退化边界

#### 结构定义

重复元素会破坏局部单调判断。例如：

```text
[2, 2, 2, 0, 1, 2]
```

当 `nums[mid] == nums[right]` 时，`mid` 和 `right` 无法告诉我们最小值在哪一侧。

#### 常见处理

```java
if (nums[mid] > nums[right]) {
    left = mid + 1;
} else if (nums[mid] < nums[right]) {
    right = mid;
} else {
    right--;
}
```

`right--` 只删除一个无法提供信息的重复端点，因此最坏复杂度可能退化为 `O(n)`。

---

## ⑤ 与其他章节的边界

| 容易混淆 | 本章结构 | 另一个结构 | 核心区别 |
|---|---|---|---|
| 旋转二分 vs 标准二分 | 局部单调 | 整体单调 | 旋转数组必须先判断哪一侧可用 |
| 找 target vs 找最小值 | `mid` 命中可丢弃 | `mid` 可能是答案 | 两者边界更新不同 |
| 旋转二分 vs 对撞 | 折半排除 | 线性收缩 | 旋转二分仍依赖半区排除 |
| 重复元素 vs 无重复元素 | 单调性可能退化 | 判断稳定 | 重复会导致最坏 `O(n)` |

---

## ⑥ 本章最容易误判的地方

- 把找最小值写成 `left <= right`。
- 在 `nums[mid] <= nums[right]` 时写 `right = mid - 1`，丢失可能答案。
- 查找 `target` 时忘记先判断 `nums[mid] == target`。
- 判断有序半边后，没有再判断 `target` 是否落在该半边。
- 忽略重复元素导致局部单调性不可判定。

---

## ⑦ 小结

旋转数组二分的统一视角是：

> 整体单调性被破坏，但每次仍要找到一个可证明排除的半边。

查找 `target` 时，关键是判断目标是否在有序半边；查找最小值时，关键是维护“最小值仍在 `[left, right]`”的不变量。两个问题边界语义不同，不能混用模板。
