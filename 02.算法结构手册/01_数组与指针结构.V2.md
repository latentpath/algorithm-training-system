# 第一章：数组与指针结构

> 核心信念：**指针移动不是技巧，而是边界不变量的维护。**
> 在顺序结构中，指针的每一次移动都必须回答：哪些区域已经确定，哪些区域仍未处理，为什么被跳过的部分不会影响答案。

---

## ① 本章解决什么结构问题？

本章处理的是线性顺序空间中的边界运动问题。

这里的线性顺序空间包括：

- 数组
- 字符串
- 链表
- 由数组下标隐式形成的链式结构

本章不以“某个双指针算法”为核心，而关注：

> 如何在顺序空间中，用有限个指针维护结构不变量，并安全地缩小、构造、划分或重连状态空间。

指针结构通常完成四类任务：

| 结构任务 | 本质 |
|---|---|
| 缩小候选空间 | 通过单调性或可排除性移动边界 |
| 原地构造答案 | 用读写边界维护已构造区域 |
| 维护多个区域 | 用多个边界划分不同语义区间 |
| 重连线性结构 | 在链表中维护入口、出口与局部段关系 |

无论表面形式如何，指针题都必须先写清楚：

- 指针分别代表什么边界
- 哪些区间已经处理完成
- 哪些区间仍然未知
- 移动某个指针时，为什么不会漏解或破坏已构造结果

---

## ② 本章的结构判断入口

以下信号说明问题可能属于数组与指针结构。

| 结构信号 | 可能对应的模型 |
|---|---|
| 两端元素共同决定当前状态 | 对撞模型 |
| 一个指针扫描，一个指针维护写入边界 | 同步追逐 / 覆盖模型 |
| 同一结构中存在不同速度的推进 | 非同步追逐模型 |
| 需要把元素原地分到不同区域 | 分区模型 |
| 多条有序序列共同推进 | 双源合并模型 |
| 链表需要拆段、反转、重连 | 重组模型 |
| 连续区间边界随条件移动 | 窗口边界模型 |

这些信号只能帮助进入结构分析，不能替代不变量证明。

例如：

- “已排序 + 两端选择”只有在移动一端可以证明排除一批候选时，才是对撞结构。
- “连续子数组”只有在窗口可以安全扩张和收缩时，才适合窗口结构。
- “原地修改数组”只有在写入区语义清晰时，才适合覆盖结构。

---

## ③ 本章的结构骨架

指针结构的通用建模流程是：

1. 定义每个指针的语义。
2. 用区间表达已处理、处理中、未处理区域。
3. 写出每次移动前后保持不变的性质。
4. 判断移动是否能安全排除、构造或转移状态。
5. 最后才落到代码骨架。

可以用如下形式描述不变量：

```text
[0, slow)      已构造答案区
[slow, fast)   已扫描但未保留/未归位区域
[fast, n)      未处理区域
```

或：

```text
[0, lt)        属于 A 区
[lt, i)        属于 B 区
[i, gt]        未判定区
(gt, n - 1]    属于 C 区
```

代码只是这种区间关系的实现。

## 1）对撞模型：从两端收缩候选空间

### 1. 结构定义

对撞模型用 `left` 和 `right` 表示当前仍可能产生答案的两端边界。

它解决的问题不是“两个指针怎么走”，而是：

> 在一个有序、单调或对称的顺序空间中，如何通过一次比较排除一侧候选。

### 2. 不变量 / 状态含义

典型状态：

```text
[0, left)          已被证明不需要再考虑
[left, right]      当前候选空间
(right, n - 1]     已被证明不需要再考虑
```

每次移动必须保持：

- 被移出候选空间的元素不会参与更优解
- 当前答案已经记录了移动前这一轮可能产生的信息
- 剩余区间仍覆盖所有未排除候选

### 3. 边界如何移动

对撞结构成立的关键是“移动依据可证明”。

例如在有序数组两数之和中：

- 当前和太小，移动 `left` 才可能增大和
- 当前和太大，移动 `right` 才可能减小和

如果无法说明移动某一侧会排除一批无效候选，就不应使用对撞模型。

### 4. 代码骨架

```java
int left = 0;
int right = nums.length - 1;

while (left < right) {
    // 先读取当前边界形成的状态，必要时更新答案。

    if (shouldMoveLeft(nums[left], nums[right])) {
        left++;
    } else {
        right--;
    }
}
```

### 5. 适用条件

- 顺序空间中存在可利用的单调性、排序关系或对称关系
- 当前两端状态足以决定至少一侧可以被排除
- 每次移动都能减少候选空间

### 6. 失效条件

- 输入无序，且移动一端无法形成可证明排除
- 当前决策依赖全局历史状态，而不是两端局部状态
- 问题需要枚举所有组合，不能通过一侧移动批量排除

失效时通常转向：

- 哈希结构
- 排序后重新建模
- 二分结构
- 搜索结构

### 7. 常见结构误判

- 只因为题目出现“两端”就使用对撞，但移动规则没有排除性。
- 把“线性收缩边界”误判为二分。对撞每次只移动一端，二分是在单调判定空间中指数级缩小范围。
- 去重时没有区分“先使用当前状态”与“先跳过重复状态”的顺序。

## 2）同步追逐模型：扫描指针与写入边界

### 1. 结构定义

同步追逐模型用一个指针扫描输入，用另一个边界维护已构造结果。

它解决的是：

> 在保持线性扫描的同时，原地构造一个满足约束的前缀区域。

常见形式是 `fast` 负责读取，`slow` 负责表示下一个写入位置。

### 2. 不变量 / 状态含义

```text
[0, slow)       已构造的有效答案区
[slow, fast)    已扫描但未进入答案区，或尚未被覆盖的区域
[fast, n)       未扫描区域
```

`slow` 的语义必须固定：

- 要么表示“下一个可写位置”
- 要么表示“最后一个有效位置”

同一份代码中不能混用。

### 3. 边界如何移动

每扫描一个 `fast`：

- 如果当前元素应该保留，写入 `slow`，再移动 `slow`
- 无论是否保留，`fast` 都继续向前扫描

核心是保证 `[0, slow)` 始终是合法答案前缀。

### 4. 代码骨架

```java
int slow = 0;

for (int fast = 0; fast < nums.length; fast++) {
    if (keep(nums, fast, slow)) {
        nums[slow] = nums[fast];
        slow++;
    }
}

// slow 表示新长度，答案区为 [0, slow)。
```

允许重复不超过 `k` 次时，状态仍然是答案前缀：

```java
int slow = 0;

for (int fast = 0; fast < nums.length; fast++) {
    if (slow < k || nums[fast] != nums[slow - k]) {
        nums[slow] = nums[fast];
        slow++;
    }
}
```

### 5. 适用条件

- 答案可以按原顺序构造成前缀
- 当前元素是否保留只依赖已构造前缀的有限信息
- 不需要在未扫描区域中回头寻找更优位置

### 6. 失效条件

- 保留当前元素会影响未来全局最优选择，无法只靠前缀状态判断
- 题目不要求保持相对顺序，交换式分区可能更合适
- 需要维护连续区间统计，而不是构造答案前缀

失效时通常转向：

- 分区模型
- 窗口结构
- 动态规划或贪心结构

### 7. 常见结构误判

- `slow` 的含义前后不一致，导致返回长度或覆盖位置错位。
- 把“构造答案前缀”与“维护当前窗口”混淆。前者答案区固定在数组前部，后者边界持续滑动。
- 对允许重复次数的问题，只比较相邻元素，而没有用答案区的第 `k` 个回看位置表达约束。

## 3）非同步追逐模型：速度差产生结构信息

### 1. 结构定义

非同步追逐模型让两个指针以不同速度在同一线性或隐式链式结构中推进。

它解决的是：

> 当结构中可能存在环、入口或中点时，如何利用速度差定位结构性质。

这里的重点不是“快慢指针模板”，而是指针运动背后的空间关系。

### 2. 不变量 / 状态含义

常见状态：

- `slow` 每轮走一步
- `fast` 每轮走两步
- 如果存在环，`fast` 会在环内追上 `slow`
- 相遇后，从头结点和相遇点同步推进，会在环入口相遇

数组也可能形成隐式链表：

```text
index -> nums[index]
```

这时移动的是映射关系，不是物理相邻下标。

### 3. 边界如何移动

判环阶段：

```text
slow = next(slow)
fast = next(next(fast))
```

入口定位阶段：

```text
slow = head
slow 和 fast 每次都走一步
```

两个阶段的运动规则不同，不能混在一起理解。

### 4. 代码骨架

```java
ListNode slow = head;
ListNode fast = head;

while (fast != null && fast.next != null) {
    slow = slow.next;
    fast = fast.next.next;

    if (slow == fast) {
        break;
    }
}

if (fast == null || fast.next == null) {
    return null; // 无环
}

slow = head;
while (slow != fast) {
    slow = slow.next;
    fast = fast.next;
}

return slow; // 环入口
```

### 5. 适用条件

- 数据可以被看作“每个位置有唯一后继”的函数图或链表
- 需要判断环、找入口、找中点或利用速度差定位结构
- 空间约束要求不能额外记录访问状态

### 6. 失效条件

- 一个节点可能有多个后继，结构已变成一般图
- 后继关系会动态变化，速度差不再稳定
- 需要找所有环或所有连通关系

失效时通常转向：

- 图遍历
- 拓扑排序
- 并查集
- 哈希记录访问状态

### 7. 常见结构误判

- 把数组中的“下一个位置”误写成 `i + 1`，而不是题目定义的映射。
- 没有先完成相遇判定，就直接进入入口定位。
- 在一般图问题中套用快慢指针，忽略了“唯一后继”这个前提。

## 4）覆盖模型：读区与写区的原地压缩

### 1. 结构定义

覆盖模型是同步追逐的一种特化。

它强调：

> 输入区和输出区位于同一个容器中，写入边界前的区域必须始终保持合法。

它常用于原地删除、原地压缩、移动元素等问题。

### 2. 不变量 / 状态含义

```text
[0, write)      已写入的合法结果
[write, read)   已读取但不属于最终前缀的旧数据
[read, n)       未读取数据
```

旧数据可以被覆盖，因为它已经被读取或不再需要。

### 3. 边界如何移动

- `read` 每次向前读取一个元素
- 当元素应进入结果时，写到 `write`
- 写入后 `write` 向前移动

### 4. 代码骨架

```java
int write = 0;

for (int read = 0; read < nums.length; read++) {
    if (shouldKeep(nums[read])) {
        nums[write] = nums[read];
        write++;
    }
}
```

### 5. 适用条件

- 结果可以覆盖原数组前缀
- 保持相对顺序通常是必要条件
- 被覆盖位置中的旧值后续不再需要

### 6. 失效条件

- 题目不要求保持相对顺序，使用两端交换可能更简单
- 需要从尾部向前写，避免覆盖尚未读取的数据
- 输出区与输入区重叠方式会破坏未读数据

典型例子是合并两个有序数组到第一个数组尾部空间中：应从后往前写，而不是从前往后覆盖。

### 7. 常见结构误判

- 认为所有原地题都是覆盖模型，忽略了分区和倒序写入。
- 写入后忘记移动 `write`，导致答案区不扩张。
- 在需要保留相对顺序的问题中使用交换，破坏顺序不变量。

## 5）分区模型：多个语义区域的边界维护

### 1. 结构定义

分区模型用多个指针维护多个连续区域，使每个区域具有明确语义。

它解决的是：

> 如何在原地扫描过程中，让元素逐步归入不同区域，并保持区域不变量。

### 2. 不变量 / 状态含义

三分区的经典表达：

```text
[0, lt)         小于目标 / A 区
[lt, i)         等于目标 / B 区
[i, gt]         未判定区
(gt, n - 1]     大于目标 / C 区
```

`i` 不是普通扫描指针，而是“当前未判定元素”的位置。

### 3. 边界如何移动

对当前 `nums[i]` 分类：

- 属于左区：与 `lt` 交换，`lt++`，`i++`
- 属于中区：`i++`
- 属于右区：与 `gt` 交换，`gt--`，但 `i` 不一定移动

原因是从右侧换来的元素仍处于未判定状态。

### 4. 代码骨架

```java
int lt = 0;
int i = 0;
int gt = nums.length - 1;

while (i <= gt) {
    if (belongsLeft(nums[i])) {
        swap(nums, lt, i);
        lt++;
        i++;
    } else if (belongsRight(nums[i])) {
        swap(nums, i, gt);
        gt--;
    } else {
        i++;
    }
}
```

二分区可以看作只有左右两个语义区域：

```java
int left = 0;
int right = nums.length - 1;

while (left <= right) {
    if (belongsLeft(nums[left])) {
        left++;
    } else if (belongsRight(nums[right])) {
        right--;
    } else {
        swap(nums, left, right);
        left++;
        right--;
    }
}
```

### 5. 适用条件

- 每个元素可以被局部判定属于哪个区域
- 题目允许原地交换
- 区域之间只要求分类正确，不一定要求保持原相对顺序

### 6. 失效条件

- 题目要求稳定排序或保持相对顺序
- 元素所属区域依赖全局最优选择，而不是局部属性
- 区域不是连续语义，无法用边界表达

失效时通常转向：

- 覆盖模型
- 稳定排序
- 计数结构
- 贪心或动态规划

### 7. 常见结构误判

- 与 `gt` 交换后直接 `i++`，跳过了换来的未判定元素。
- 只记住交换代码，却没有写清楚每个区间的语义。
- 在要求稳定性的题目中使用分区交换，破坏相对顺序。

## 6）双源合并模型：多条有序流的同步推进

### 1. 结构定义

双源合并模型同时维护两条或多条有序来源的当前位置，每次选择当前应进入结果的来源并推进。

它解决的是：

> 如何在多个已排序或已按规则推进的序列中，保持输出结构的有序性或最优性。

### 2. 不变量 / 状态含义

两路合并的典型状态：

```text
A[0, i)      已从 A 消费
B[0, j)      已从 B 消费
output       已构造且满足目标顺序
```

每次只比较当前边界元素，输出较优者。

### 3. 边界如何移动

- 选择 `A[i]` 时，推进 `i`
- 选择 `B[j]` 时，推进 `j`
- 任一来源耗尽后，处理另一个来源的剩余部分

如果输出区与输入区重叠，需要重新判断写入方向。

### 4. 代码骨架

```java
int i = 0;
int j = 0;

while (i < m && j < n) {
    if (A[i] <= B[j]) {
        take(A[i]);
        i++;
    } else {
        take(B[j]);
        j++;
    }
}

while (i < m) {
    take(A[i]);
    i++;
}

while (j < n) {
    take(B[j]);
    j++;
}
```

重叠写入时常从后往前：

```java
int i = m - 1;
int j = n - 1;
int write = m + n - 1;

while (j >= 0) {
    if (i >= 0 && A[i] > B[j]) {
        A[write--] = A[i--];
    } else {
        A[write--] = B[j--];
    }
}
```

### 5. 适用条件

- 每个来源内部已经有序，或推进顺序本身具有单调性
- 当前来源头部足以决定下一步输出
- 输出结果可由局部最优选择逐步构造

### 6. 失效条件

- 来源内部无序，当前头部无法代表该来源的最优候选
- 需要跨来源进行全局组合搜索
- 来源数量很多且需要动态选择全局最小或最大

失效时通常转向：

- 排序
- 堆与优先级结构
- 图上的最短路 / 多源扩展

### 7. 常见结构误判

- 忽略尾部收尾，导致某一来源剩余元素丢失。
- 在重叠数组中从前往后写，覆盖尚未读取的数据。
- 多路合并时仍手写多指针比较，复杂度和代码稳定性都变差；应考虑堆。

## 7）链表重组模型：拆段、反转与重连

### 1. 结构定义

链表重组模型不依赖下标，而依赖节点之间的 `next` 关系。

它解决的是：

> 在无法随机访问的线性结构中，如何通过维护局部入口和出口，完成拆段、反转、交换或旋转。

链表题的核心不是遍历，而是重连顺序。

### 2. 不变量 / 状态含义

常见边界：

- `dummy`：统一处理头节点变化
- `prev`：待操作区间前一个节点
- `start`：待操作区间原头部，操作后可能成为尾部
- `next`：待操作区间之后的第一个节点

每次操作后必须保持：

- 已处理前缀已经连接完整
- 当前操作段有明确入口和出口
- 后续未处理段没有丢失

### 3. 边界如何移动

链表局部重组通常分四步：

1. 定位操作段前驱 `prev`
2. 保存操作段之后的节点
3. 在局部段内改变 `next`
4. 将前驱、局部段、新尾部、后续段重新连接

任何一步丢失引用，都可能导致链表断裂或成环。

### 4. 代码骨架

```java
ListNode dummy = new ListNode(0);
dummy.next = head;

ListNode prev = dummy;

// 1. prev 定位到待操作区间之前
// 2. 识别区间头 start 和区间后继 next
// 3. 局部反转或重排
// 4. prev.next 指向新头，旧头作为新尾连接 next

return dummy.next;
```

局部反转常见骨架：

```java
ListNode prev = null;
ListNode cur = head;

while (cur != null) {
    ListNode next = cur.next;
    cur.next = prev;
    prev = cur;
    cur = next;
}

return prev;
```

### 5. 适用条件

- 结构是单向或双向链表
- 问题要求改变节点连接关系，而不是只读取值
- 操作可以分解为有限段的拆分与重连

### 6. 失效条件

- 问题本质是按值查询、排序或随机访问
- 需要频繁按下标定位节点
- 重组规则依赖全局最优，而不是局部段操作

失效时通常转向：

- 数组化处理
- 栈或递归
- 排序结构
- 额外哈希映射

### 7. 常见结构误判

- 不使用 `dummy` 处理头节点变化，导致头部边界分支过多。
- 改 `next` 前没有保存后继节点，丢失剩余链表。
- 局部反转后没有明确“新头”和“新尾”，重连时形成断链或环。

## 8）窗口边界模型：连续区间的左右边界维护

### 1. 结构定义

窗口边界模型用 `left` 和 `right` 表示一个连续区间。

在本章中，它只作为指针运动模型出现；窗口内统计量、前缀和替代关系和失效条件会在第二章系统展开。

它解决的是：

> 如何让一个连续区间在扩张和收缩之间保持约束。

### 2. 不变量 / 状态含义

```text
[left, right]   当前窗口
[0, left)       已经不再作为当前窗口起点考虑
(right, n - 1]  尚未进入窗口
```

窗口结构的关键不是两个指针，而是窗口内状态是否能被增量维护。

### 3. 边界如何移动

典型变长窗口：

```java
int left = 0;

for (int right = 0; right < nums.length; right++) {
    add(nums[right]);

    while (!valid()) {
        remove(nums[left]);
        left++;
    }

    updateAnswer(left, right);
}
```

### 4. 适用条件

- 目标对象是连续子数组或连续子串
- 窗口扩张和收缩时，状态可以增量维护
- 约束具有足够的单调性，使得收缩是安全的

### 5. 失效条件

- 存在负数导致区间和约束不再随边界单调变化
- 约束依赖非局部信息，无法通过加入/移除一个元素维护
- 目标是子序列而不是连续区间

失效时通常转向：

- 前缀和 + 哈希
- 前缀和 + 单调队列
- 动态规划
- 二分答案

### 6. 常见结构误判

- 只看到“连续”就使用窗口，但无法证明收缩安全。
- 把子序列问题误判为子数组窗口问题。
- 在窗口内维护最大值或最小值时，仍用普通变量而不是单调队列。

## ⑤ 与其他章节的边界

| 容易混淆 | 本章结构 | 另一个结构 | 核心区别 |
|---|---|---|---|
| 对撞 vs 二分 | 线性移动左右边界 | 在单调判定空间中折半 | 对撞依赖局部排除，二分依赖全局判定单调性 |
| 窗口边界 vs 前缀结构 | 动态维护连续区间状态 | 用两个前缀表达任意区间 | 窗口要求可安全收缩，前缀不要求局部可维护 |
| 覆盖 vs 分区 | 构造合法答案前缀 | 维护多个分类区域 | 覆盖通常保持顺序，分区通常允许交换 |
| 快慢指针 vs 图判环 | 每个位置只有唯一后继 | 一般图可能有多个后继 | 快慢指针适合函数图，一般图需要 visited / recursion stack |
| 双源合并 vs 堆 | 两个或少数有序来源 | 多来源动态优先级 | 来源多且需要反复取全局最优时，堆更稳定 |
| 链表重组 vs 递归结构 | 改变线性 next 关系 | 汇总子问题信息 | 重组关注连接关系，递归关注信息返回 |

---

## ⑥ 本章最容易误判的地方

- 没有先定义指针语义，就直接写移动规则。
- 只看题面关键词，不验证移动是否能保持不变量。
- 把“局部可移动”误认为“全局正确”。指针每移动一步，都必须能解释被排除或被构造的区域。
- 混淆不同模型中的同名指针。例如 `slow` 在覆盖模型中是写入边界，在快慢指针中是运动速度较慢的当前位置。
- 忽略顺序要求。需要保持相对顺序时，分区交换往往不适用。
- 忽略写入覆盖风险。输入区和输出区重叠时，必须判断从前写还是从后写。
- 把窗口当作普通双指针。窗口的核心是统计不变量，不是两个边界本身。

## ⑦ 小结

数组与指针结构的统一视角是：

> 在顺序空间中，用指针表达边界，用区间表达状态，用不变量证明移动安全。

本章的关键不是记住某个模板，而是能在写代码前回答三件事：

1. 每个指针代表什么结构边界？
2. 指针两侧的区域分别处于什么状态？
3. 这次移动为什么不会漏解、错构或破坏已有结果？

当这些问题无法被清楚回答时，说明当前题目可能不适合本章结构，需要迁移到前缀、二分、图、堆、动态规划或搜索等其他结构。
